# 个人作品集文案优化方案

## 1. 目标与依据

本方案基于 [`resume-zhuoqi.html`](S:/Personal%20info/CV/resume-zhuoqi.html) 与当前作品集代码整理。本轮**只新增方案文档，不修改页面代码**。

目标不是把页面堆成技术名词墙，而是让招聘方在每个屏幕都能快速看懂：

1. 我负责过什么级别、什么类型的系统；
2. 我怎样拆解架构、性能、实时交互和 AI 系统问题；
3. 这些工作产生了哪些可核验的结果。

网站的主叙事会调整为：**有 9 年企业级交付经验的高级全栈工程师，以前端架构为根基，能把复杂业务、实时数据和 Agent 能力交付为可维护的生产系统。**

## 2. 全站文案原则

| 原则                   | 调整方式                                                                                       |
| ---------------------- | ---------------------------------------------------------------------------------------------- |
| 先结论，后技术         | 先说交付对象或工程难题，再补充技术栈；不以一串框架名开头。                                     |
| 每条都能追溯           | 只使用简历中已有的职责、技术与量化结果；不新增未经证实的性能、成本或业务指标。                 |
| 用“动作 + 机制 + 结果” | 例如“以虚拟滚动处理万级资产列表，渲染耗时降低 60%、内存降低 50%”。                             |
| 统一术语               | 全站统一使用“Agent 编排、混合检索、流式交互、权限边界、工程体系”等词，首次出现时交代业务含义。 |
| 保持易读               | 英文技术名词保留，但相邻一句中文说明其作用；单个卡片正文控制在 2–3 句，指标区使用短句。        |

建议保留、反复出现的硬证据：**9 年+交付、6 人团队 / 5 个项目 / 20 万+行代码、TTI -40%、60fps、API 成功率 98%、首屏 ≤1.5s、万级列表渲染 -60% / 内存 -50%、开发效率 +60%、Bug 率 -70%、页面 5s → 1s 内、jQuery → React 后渲染 4 倍、审计报告生成 3 倍。**

#### 首屏配图重组与生成方向

现有五张图都是统一的黑白铅笔写实风，人物识别度很好；其中正面头像可继续用于 01，但“办公室、两张生活照、羽毛球”目前更像人物日常相册。建议重组为 **4 张技术主题图 + 1 张生活合成图**，全部继续使用同一人物、同一铅笔线稿、同一纸张底色，避免视觉断裂。

| 卡片                         | 图片职责                     | 建议画面                                                                                                                           | 生成时要避开                                                            |
| ---------------------------- | ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| 01 `ENGINEER_PROFILE`        | 建立人物识别                 | 保留现有正面头像，裁成胸像；背景只保留极淡的方格纸与 1–2 个抽象界面轮廓。                                                          | 不增加满屏代码、电脑或夸张的“黑客”元素。                                |
| 02 `SYSTEMS_AND_PERFORMANCE` | 显示系统与性能能力           | 人物坐在双屏工作台前，一侧是抽象的业务流程图（租户 / 权限 / 审批 / 对账四个无文字模块），一侧是平滑上升的性能曲线与列表窗口。      | 不让模型生成可读代码、伪造数据或密集的屏幕 UI；真实指标由网页文字叠加。 |
| 03 `AGENT_ENGINEERING`       | 显示 Agent 工程而非“AI 氛围” | 人物站在白板或透明面板前，面板呈现四层结构：文档卡片 → 检索节点 → Agent 状态图 → 工具节点；用连线、方块和少量酸绿 / 电紫点缀。     | 不用霓虹城市、机器人、发光大脑、赛博黑客房间等泛 AI 意象。              |
| 04 `DELIVERY_SYSTEM`         | 显示工程化与团队交付         | 一张“组件装配台”：人物正在把表单、表格、状态机、测试勾选和部署管线等抽象模块拼入一张产品蓝图；画面偏平面设计稿，不必出现其他人物。 | 不生成握手、会议室合影、看板便利贴墙等泛管理画面。                      |
| 05 `OFFLINE_MODE`            | 用一个画面带过生活           | 同一张竖幅内做上下或左右拼贴：小幅滑雪剪影 + 家中与两只猫的安静场景；面积与视觉对比低于前四张。                                    | 不再单独使用羽毛球和猫的完整卡片，也不要让生活画面出现在前四张技术卡。  |

所有技术图都应为“**人物 + 可辨识的工程结构**”，而不是“人物在敲电脑”。由于生成模型很难可靠地产生文字和代码，图中只生成几何模块、无文字图表、连线和留白；`React`、`RAG`、`60fps` 等信息由网页层的真实排版呈现。

可直接用于生成工具的共用约束：`same male engineer as reference, black graphite pencil illustration on off-white textured paper, confident but understated, crisp architectural linework, neo-brutalist composition, portrait 4:5, generous negative space, no text, no logos, no watermark, no fake code, no cyberpunk neon`。每张再追加上表的“建议画面”。

### 3.3 技术能力雷达

雷达图的分数是自评，技术可信度弱于项目证据。建议继续保留互动形式，但把数字的视觉权重降低（或改成“能力覆盖度”），让右侧内容成为主证据。维度名称也从抽象的“前端 / 后端”升级为招聘方更容易判断岗位匹配度的工程能力。

| 现有维度 | 建议维度               | 核心技能文案                                                                    | 实战证据文案                                                                                                                              |
| -------- | ---------------------- | ------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| 前端     | **前端架构与性能**     | `React 18+ · Next.js App Router · TypeScript · Vue · TanStack Query · Zustand`  | `为多租户 SaaS 设计混合渲染、缓存与跨模块状态策略；核心页面 TTI 降低 40%。在万级资产列表中使用虚拟滚动，渲染耗时降低 60%、内存降低 50%。` |
| 后端     | **服务与数据协同**     | `Node.js · Next.js API Routes · FastAPI · PostgreSQL · REST · Docker · nginx`   | `以 Node.js + PostgreSQL 交付审批流程引擎；在高交易量场景封装并发 ≤10、请求去重与指数退避重试的 Fetch Queue，API 成功率 98%。`            |
| AI建设   | **Agent 与 RAG 工程**  | `LangChain · LangGraph · MCP · Function Calling · Chroma · SSE · RAGAS`         | `独立完成文档导入、向量嵌入、语义 + BM25 混合检索到 LLM 生成的 RAG 管线；以 RAGAS 量化检索质量，并将能力接入业务操作流。`                 |
| 团队管理 | **工程质量与交付领导** | `Design Token · 组件库 · Jest / Vitest · ESLint · GitLab CI/CD · Agile / Scrum` | `带领 6 人团队交付 5 个项目、累计 20 万+行代码；建设组件库和规范体系，开发效率提升 60%，Bug 率降低 70%。`                                 |

右侧面板的固定栏目建议改为：

- `ENGINEERING SCOPE`：技术边界与关键组件；

- `VERIFIED OUTCOME`：只展示简历中有来源的结果指标。

这样可避免当前 `Core Skills / State / Data / Field Experience` 出现“工具清单很多、技术决策不够明显”的感觉。

### 3.5 AI 能力落地区（Agent Sandbox）

当前文案面向非技术读者比较友好，但过于泛化，无法体现已做过 LangGraph、RAGAS、SSE、MCP 等工程工作。建议保留“业务可用、边界可控”的叙事，但让技术链路更清晰。

**区块标题**

> 把 Agent 做成可控、可评估、可接入业务流程的系统。

**区块引言**

> AI 能力的难点不只是接入模型，而是把上下文、知识、工具权限、流式交互和质量评估组织成一条可运行的工程链路。我关注它能否在真实流程中稳定地帮助用户完成任务。

| 节点 | 标题                     | 摘要                                                           | 展开后“怎么做”                                                                                                          | 展开后“带来的价值”                                                    |
| ---- | ------------------------ | -------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| 01   | **任务与边界建模**       | 先定义用户意图、业务状态与人工确认点，再设计 Agent 路由。      | `将审批条件、租户权限和会话上下文映射为状态；高风险动作保留 Human-in-the-loop 确认。`                                   | `避免 Agent 越权执行，使自然语言入口仍遵守既有流程、权限与审计要求。` |
| 02   | **知识检索与质量评估**   | 回答必须来自可检索、可验证的企业知识，而不是模型猜测。         | `文档导入后进入 Chroma 向量库，以语义 + BM25 混合检索召回；Python 脚本结合 RAGAS 量化检索质量。`                        | `让答案更贴近租户专属政策与业务规则，且可以持续定位与优化检索问题。`  |
| 03   | **编排、工具与操作闭环** | Agent 不只回答问题，也可以在权限边界内协助查询和发起业务操作。 | `使用 LangChain / LangGraph 组织推理与工具调用，以 MCP / Function Calling 接入系统能力；自然语言请求转为可确认的操作。` | `缩短跨页面操作路径，同时保留用户确认与系统权限控制。`                |
| 04   | **流式交互与持续迭代**   | 流畅的 Agent 产品还需要可感知的响应过程和上线后的验证机制。    | `以 SSE 构建多轮消息流、上下文同步与前端状态管理；结合命中情况、异常反馈与评估结果迭代知识和提示策略。`                 | `用户看到的是可继续操作的过程，而非一次性、不可解释的模型输出。`      |

底部三项小结建议替换为：

- `业务助手`：`将自然语言意图映射为带上下文的查询与操作建议。`
- `知识问答`：`通过混合检索与评估链路，提供可验证的业务知识回答。`
- `流程协同`：`将工具调用嵌入权限、确认与审计边界，形成可控闭环。`

### 3.7 字体与排版优化方向

我已查看当前的实际样式与组件结构：页面使用 **Syne 800** 作为标题、**Space Mono** 作为正文，并且 `main` 与大量段落默认套用 `font-mono`。视觉上有鲜明的控制台感，但用于中文履历正文会造成三个问题：中文实际回退为系统字体、长段信息密度过高、所有内容的字面“重量”趋于一致。

| 层级           | 当前状态                                                  | 建议调整                                                                                                                     |
| -------------- | --------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| 中文正文       | 大量使用 `Space Mono`，中文由浏览器回退，阅读质感不可控。 | 采用一款稳定的中文无衬线体作为正文与数据说明，例如 `Noto Sans SC` 或已具备授权的中文字体；正文不再全局使用等宽字体。         |
| 英文代码标签   | Syne 与 Space Mono 已有辨识度。                           | Syne 只用于短标题和数字；Space Mono 只用于 `RAG`、`SSE`、指标标签、时间轴等短内容。                                          |
| 标题字距       | 多处 `tracking-tighter`，中英混排且标题较长时容易拥挤。   | 中文标题控制在约 `-0.02em`，不再用极紧字距；长公司名与项目名允许自然换行。                                                   |
| 正文行高与字宽 | 当前正文多为 `text-base`，并排卡片内的等宽中文会显得紧。  | 正文使用 16–18px、行高 1.7–1.85；文字区最大宽度约 34–40 个中文字符，长技术栈拆为标签或独立一行。                             |
| 信息层级       | 标题、技术栈、说明与指标常以同一种“终端式”字形出现。      | 固定为“项目标题（sans）→ 一句系统摘要（sans）→ 机制标签（mono）→ 指标短句（sans / mono 数字）”四级，阅读重心落在项目和结果。 |

布局将保留现有的新粗野纸张与硬边框语言，但会调整信息节奏：

1. 首屏卡片从“图片与长段文字并置”改为“图片承担情绪与主题、文字承担工程判断”，每卡不超过 75 个中文字符；
2. 雷达图分数弱化，右侧内容增加“技术取舍 / 结果”层级，避免自评数字抢走项目证据；
3. Dossier 的长公司名和项目名采用两行上限，四个证据格使用短标签 + 一句事实，避免卡片在中等屏幕被挤满；
4. AI 区保持深色反差，但正文改用中文 sans，等宽字体只保留节点编号和协议名，减少“终端拟态”对信息的干扰；
5. 现有图片统一改为 4:5 竖幅并在同一位置切换，避免不同人物姿势与视觉重心导致滚动时跳动。

### 3.8 页脚

当前 `Designed and Built by Sonny & AI.` 只表达了视觉制作，无法收束技术定位。建议改为不含联系方式的技术落点：

> `蔡璐阳（Sonny）｜企业级系统 · 实时数据 · AI Agent`
>
> `Built with Next.js · React · TypeScript · © {year}`

生活信息不再放在页脚；它仅在首屏最后一张 `OFFLINE_MODE` 卡中出现一次。

## 5. 实施顺序与验收

1. 先更新 `layout.tsx`、首屏卡片、配图和页脚，删除所有联系方式，确保首屏在 15 秒内传递岗位定位与证据。
2. 更新雷达维度和 Dossier 数据，统一以“机制 + 指标”说明能力。
3. 更新 Agent Sandbox 与作品区，让 AI / 个人项目文案与简历一致。
4. 落地中文字体与四级信息层级，并做一次桌面与移动端文字溢出检查；中文正文优先使用 2–3 句，数据点使用短句。
